Development Guidelines

Development Overview

The oscms-codecs-bridge library follows a clean and consistent development style adapted from the Google C++ Style Guide. The C codebase is formatted with clang-format and structured to be readable, maintainable, and portable.

C Language Standard and Tooling

  • Standard: C11 with POSIX extensions where applicable.
  • Compiler: GCC 11+ or Clang 12+.
  • Build system: CMake 3.15+.
  • Formatting: clang-format with project-provided .clang-format file.

Header Files

Self-Contained Headers

Every header file must be self-contained (able to compile on its own). If a header declares types or functions that depend on other headers, it must include those dependencies. All public headers in include/public/ must compile in isolation.

The #define Guard

All headers use traditional include guards with a consistent naming pattern derived from the full path:

#ifndef OSCMS_INCLUDE_PUBLIC_PROTOCOL_BRIDGE_H
#define OSCMS_INCLUDE_PUBLIC_PROTOCOL_BRIDGE_H

// declarations

#endif  // OSCMS_INCLUDE_PUBLIC_PROTOCOL_BRIDGE_H

Include Ordering

To minimize hidden dependencies, includes are ordered as follows:

  1. Related header (for .c files)
  2. C system headers
  3. Project headers
  4. Third-party library headers (e.g., asn1c, OpenSSL)

Scoping

Namespaces

C does not have namespaces, so function and type names use prefixes to organize scope:

  • All public API functions and types are prefixed with Oscms or oscms_.
  • Internal symbols are either declared static or prefixed with the module name to avoid symbol collisions (e.g., codec_internal_parse_point()).

Internal Linkage

Helper functions, constants, and types used only within a single translation unit should be declared static to limit their scope:

static int parse_internal_type(const uint8_t *buffer) {
    // ...
}

Local and Static Variables

Declare variables in the smallest scope possible, ideally at first use. Static variables are used sparingly and never for mutable state unless explicitly required by an initialization context.

Structs

Opaque vs. Exposed Structs

  • Opaque types are forward-declared in public headers, with definitions hidden in private headers or source files. This ensures ABI stability and encapsulation.
  • Exposed types are fully defined in public headers when their layout must be known to callers (e.g., OscmsSequence).
// Opaque (declared but not defined publicly)
typedef struct OscmsIeee1609Dot2Data OscmsIeee1609Dot2Data;

// Exposed (definition visible)
typedef struct {
    size_t count;
    void **data;
} OscmsSequence;

Struct Initialization

Initialize structs using designated initializers when possible to make initialization explicit and maintainable:

OscmsPoint point = {
    .x = 0x01,
    .y = 0x02
};

Naming Conventions

Category Convention Example
Functions snake_case oscms_spdu_construct()
Variables snake_case octet_buffer
Types PascalCase OscmsOctetBuffer
Constants UPPER_SNAKE_CASE OSCMS_MAX_BUFFER_SIZE
Enums PascalCase (type), UPPER_SNAKE_CASE (values) OscmsHashAlgorithm, OSCMS_HASH_SHA256

Extensibility Considerations

The codec abstraction layer is designed to remain independent of asn1c or any specific ASN.1 transpiler. When designing new abstractions:

  • Do not expose transpiler-generated struct layouts in public headers.
  • Use opaque types and accessor functions to isolate internal representation changes.
  • Document invariants and preconditions in header comments to guide future maintainers.
