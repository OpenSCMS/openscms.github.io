Code Structure

Rust Workspace Organization

The OpenSCMS repository is organized as a Rust workspace composed of multiple independent service crates, each implementing a specific SCMS role defined by IEEE 1609.2.1. The repository structure reflects a deliberate design choice to align code organization with system responsibilities, deployment boundaries, and trust separation.

Rather than a monolithic codebase, OpenSCMS adopts a service-per-crate model, where each SCMS component is compiled, deployed, and scaled independently.

At the top level, the repository is a Cargo workspace that groups all SCMS services and shared libraries:

openscms/
├── aca/        # Authorization Certificate Authority service
├── eca/        # Enrollment Certificate Authority service
├── ra/         # Registration Authority service
├── la/         # Linkage Authority services
├── cam/        # Certificate Access Manager
├── dc/         # Device Configuration service
├── scmscommon/ # Shared Rust utilities and abstractions
├── certs/      # Certificates and keys required at runtime
├── deployment/ # Helm charts and Kubernetes manifests
├── docker/     # CI and build container definitions
├── scripts/    # Helper and bootstrap scripts
└── oscms_bridge/
    └── oscms-codecs-bridge/ # C cryptographic and protocol engine

Each directory under the workspace root corresponds to an independent Rust crate (except certs, deployment, docker and scripts). Most of these crates (e.g., ra/, eca/, aca/, la/, cam/, dc/) are service crates that produce a single SCMS service binary, implementing a specific SCMS role and intended to be deployed as an independent microservice.

In addition to service crates, the workspace also includes shared library crates. Both scmscommon and oscms_bridge are Rust library crates (crate-type = lib) reused across multiple services. While scmscommon provides common Rust-level utilities and infrastructure abstractions, oscms_bridge acts as the Rust integration layer for the C-based oscms-codecs-bridge, encapsulating all Rust-C interoperability.

In addition to Rust crates, the repository includes a small set of directories dedicated to environment management and operational concerns rather than application logic. The certs/ directory contains the certificates and keys required at runtime by SCMS services; deployment/ provides Helm charts and Kubernetes manifests used for deployment; scripts/ contains helper scripts for bootstrapping and automation tasks; and docker/ includes container definitions used for CI and build pipelines.

Internal Structure of a Service Crate

Although each service implements a different SCMS role, their internal structure follows a common pattern, which simplifies navigation and onboarding:

  • main.rs: Service entry point, responsible for startup, configuration loading, and HTTP server initialization.
  • endpoint.rs: Definition of REST or binary endpoints exposed by the service.
  • core_logic/: Implementation of the service's domain-specific logic, including certificate workflows and protocol handling.
  • entities/: Data models representing persisted or derived SCMS entities.
  • persistence/: Database access layer and query abstractions.
  • migrator/: Schema migrations required by the service.

This separation enforces a clear distinction between transport and API concerns, protocol and cryptographic logic, and persistence and state management.

Role of scmscommon

The scmscommon crate is a Rust library crate that contains functionality shared across multiple SCMS services. This includes common configuration handling, shared data types, error abstractions, and utility routines that are orthogonal to any specific SCMS role.

By centralizing this functionality in a dedicated crate, OpenSCMS avoids code duplication while maintaining strict service boundaries. Importantly, scmscommon does not implement SCMS business logic or protocol workflows, ensuring that each service crate remains responsible only for its own role as defined by the standard.

The oscms_bridge Rust Crate

The oscms_bridge crate is a Rust library crate that provides a safe and idiomatic Rust interface to the C-based oscms-codecs-bridge. It serves as the explicit language and responsibility boundary between Rust services and the low-level cryptographic and protocol engine.

This crate contains a build.rs script responsible for:

  • Compiling the C sources of the oscms-codecs-bridge
  • Linking the resulting static or shared library into the Rust build
  • Automatically generating Rust bindings for the C API using the bindgen tool

By generating bindings at build time, oscms_bridge ensures that the Rust view of the C API remains consistent with the underlying implementation, while preserving Rust's safety guarantees through carefully designed wrapper types and ownership boundaries.

The oscms_bridge crate also contains the oscms-codecs-bridge subrepository, which is itself a fully independent C project. This C component is structurally and conceptually isolated from the Rust workspace and implements the core cryptographic and IEEE 1609.2.1 protocol logic of OpenSCMS. The oscms-codecs-bridge Overview for more details.

Dependency Model and Service Boundaries

From a Rust perspective, service dependencies are intentionally minimal and explicit. Basically, each service depends on the oscms_bridge Rust crate, which in turn manages compilation, linking, and binding generation for the underlying C-based oscms-codecs-bridge for cryptographic processing and IEEE 1609.2.1 SPDU handling. Also, no Rust service directly depends on another service's crate.

Inter-service coordination occurs exclusively through well-defined network APIs and asynchronous messaging, not through shared code or in-process calls. This enforces deployment-time isolation and mirrors real-world SCMS trust boundaries.
