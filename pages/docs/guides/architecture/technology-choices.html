Architecture

Technology Choices

Rust Backend

Rust is used for all SCMS services due to its strong guarantees around memory safety, concurrency, and performance properties that are essential for security-critical, highly concurrent systems handling sensitive cryptographic material.

An additional and decisive factor in this choice is Rust's clean and explicit integration model with C through Foreign Function Interfaces (FFI). This allows OpenSCMS to safely and efficiently interface with the oscms-codecs-bridge, a low-level C library that implements the core cryptographic and IEEE 1609.2.1 protocol logic. Rust bindings provide a clear boundary between high-level service orchestration and low-level cryptographic operations, enabling strong type safety, well-defined ownership semantics, and controlled memory management across the language boundary.

C Cryptographic Bridge

The oscms-codecs-bridge is implemented in C to provide fine-grained control over memory layout, cryptographic operations and libraries, and ASN.1 processing. The bridge library is codec-agnostic and abstracts the underlying ASN.1 transpiler behind a stable API, avoiding dependency on proprietary tooling.

Microservices and Messaging

The IEEE 1609.2.1 standard defines client vs SCMS interactions primarily through REST-based endpoints, making a microservice-oriented architecture a natural and practical design choice for OpenSCMS. By mapping each logical SCMS role to an independent service, the system closely mirrors the structure and interaction model prescribed by the standard.

Task parallelism is achieved through asynchronous messaging, enabling services to decouple request ingestion from computationally expensive cryptographic and certificate generation workflows. This design allows OpenSCMS to process large volumes of certificate requests without blocking synchronous API endpoints.

This approach is particularly important for high-demand components such as the Registration Authority (RA), where V2X clients frequently request batches of authorization or pseudonym certificates to support credential rotation. By offloading long-running tasks to message queues, the RA can scale horizontally, handle burst traffic efficiently, and maintain predictable response behavior even under heavy load.

This architecture ensures that no single component can compromise system-wide privacy guarantees while remaining practical for real-world deployments.
