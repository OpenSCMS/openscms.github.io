Setup & Deployment

Overview

This section describes how OpenSCMS can be deployed and executed in different environments, ranging from manual execution of individual components to a fully unified deployment of the complete SCMS ecosystem.

Each SCMS component in OpenSCMS (e.g., RA, ECA, ACA, LA) is built as an independent service and can, in principle, be deployed manually as a standalone process. In this model, the operator is responsible for ensuring that all external services required by the component are already deployed, properly configured, and reachable. These external dependencies include, but are not limited to:

  • A relational database (MySQL) for persistent state
  • A message queue service (RabbitMQ), required primarily by the Registration Authority (RA)

The deployment, installation, and configuration of these external services are considered out of scope for this document.

To simplify deployment and reduce operational complexity, OpenSCMS also provides a unified deployment approach based on Docker, Kubernetes, and Skaffold. This approach deploys the entire OpenSCMS ecosystem in a coordinated manner, including all SCMS service components, dedicated database instances for each component, and a RabbitMQ instance used by the RA.

This unified deployment mechanism is the recommended way to run OpenSCMS, particularly for development, testing, and evaluation purposes, as it ensures a consistent environment and minimizes configuration errors.

Build Dependencies

OpenSCMS relies on a combination of Rust tooling, native system libraries, and container-orchestration utilities. The required dependencies can be divided into two categories: dependencies required to build and run individual SCMS components, and dependencies required for unified ecosystem deployment.

Core Build and Runtime Dependencies

The following tools and libraries are required to build and execute individual OpenSCMS components (e.g., RA, ECA, ACA) manually:

  • Rust toolchain (rustc, cargo, rustfmt, clippy)
  • CMake
  • libclang-dev
  • llvm-dev
  • libzip-dev
  • libssl-dev

These dependencies are required to compile both the Rust service crates and the C-based oscms-codecs-bridge, as well as to generate Rust bindings via bindgen.

Unified Deployment Dependencies

For deploying the complete OpenSCMS ecosystem using the recommended unified approach, the following additional tools are required:

  • helm
  • kubectl
  • kubeseal
  • minikube
  • skaffold
  • sea-orm-cli (for database migrations and schema management)

OpenSCMS uses binenv to manage consistent versions of Kubernetes-related tooling. To install binenv, follow the official instructions:

wget -q -O binenv https://github.com/devops-works/binenv/releases/download/v0.19.0/binenv_linux_amd64
chmod +x ./binenv
./binenv update
./binenv install binenv
rm ./binenv
echo 'export PATH=$HOME/.binenv:$PATH' >> $HOME/.bashrc
echo 'source <(binenv completion bash)'
exec $SHELL

Once binenv is installed, install all required tools. The exact versions are pinned in the .binenv.lock file at the repository root:

binenv install -l

From this point on, entering the repository directory will automatically expose the required tools in the shell PATH.

A complete reference environment, including all required dependencies, is also documented in docker/lgscms-ci.dockerfile.

Component Certificates and Keys

Before deploying OpenSCMS, a complete and consistent set of certificates and cryptographic keys must be generated for all internal SCMS components and external trust anchors. While certificate generation is formally outside the scope of the OpenSCMS runtime, it is a mandatory prerequisite for any functional deployment.

All SCMS services expect certificates and keys to be available locally at startup, following a predefined directory structure under the certs/ folder. Missing or inconsistent cryptographic material will prevent services from starting correctly.

To simplify certificate and key generation for development and testing deployments, OpenSCMS provides the scms_manager utility. This tool is located within the C-based oscms-codecs-bridge project:

openscms/oscms_bridge/oscms-codecs-bridge/scms_manager

The scms_manager generates all cryptographic payloads required by OpenSCMS services, including component certificates, public and private keys, and protocol-specific artifacts. It also produces external trust material used by V2X clients, such as elector certificates and Root CA keys.

Generating Certificates

From the repository root, the following script can be used to invoke the scms_manager and populate the certs/ directory automatically:

sh scripts/run_tests_scms_manager.sh

After execution, the certs/ directory is populated with the expected structure and files:

├── aca
│   ├── acaCertificate.coer
│   ├── acaPrivateKey.dat
│   └── acaPublicKey.dat
├── ctl
│   ├── ccf.coer
│   └── ctl.coer
├── eca
│   ├── ecaCertificate.coer
│   ├── ecaPrivateKey.dat
│   └── ecaPublicKey.dat
├── ica

In addition to internal SCMS component certificates, the scms_manager generates external trust artifacts that are not consumed directly by OpenSCMS services but are essential for V2X clients.

Manual Deployment Example: Registration Authority

This subsection illustrates a manual deployment of a single OpenSCMS component using the Registration Authority (RA) as a concrete example. The same general procedure applies to other SCMS services, with variations only in required external dependencies and configuration parameters.

Manual deployment is primarily intended for development, debugging, and advanced experimentation scenarios. For most use cases, the unified deployment approach described later in this section is strongly recommended.

Prerequisites and Environment Setup

Before starting the RA service manually, the following conditions must be met:

  • All build dependencies described earlier in this section are installed
  • The certs/ directory is populated with valid certificates and keys
  • A MySQL instance is running and reachable by the RA service
  • A RabbitMQ instance is running and reachable by the RA service

The deployment, installation, and configuration of MySQL and RabbitMQ are outside the scope of this document.

For this example, the following connection strings are assumed:

MYSQL_CONNECTION_STRING=mysql://username:password@mysql:3306
RABBITMQ_CONNECTION_STRING=amqp://rmqadmin:password@scmsrabbitmq:5672

In addition to the global deployment parameters, the RA requires database and messaging configuration to be provided via environment variables. At a minimum, the following variables must be defined:

MYSQL_USERNAME=username
MYSQL_PASSWORD=password
MYSQL_HOST=mysql
MYSQL_PORT=3306
MYSQL_CONNECTION_STRING=mysql://username:password@mysql:3306

RABBITMQ_CONNECTION_STRING=amqp://rmqadmin:password@scmsrabbitmq:5672

Building and Running the RA Service

Once the environment is correctly configured, the RA service can be built and started.

From the repository root, build the complete Rust workspace:

cargo build

This step compiles all Rust service crates as well as the C-based oscms-codecs-bridge, and generates the required Rust bindings via bindgen. Build failures at this stage typically indicate missing system dependencies or misconfigured toolchains.

To start the Registration Authority service, change into the RA crate directory and run:

cd openscms/ra
cargo run

Upon startup, the RA service will load its certificates and keys, establish connections to the configured MySQL database and RabbitMQ broker, and expose its REST endpoints on the configured port.

Startup failures usually result from missing certificates, incorrect environment variables, or unavailable external services.

Using Docker and Kubernetes

While manual deployment is useful for development and debugging, the recommended way to run OpenSCMS is through its unified deployment mechanism based on Docker, Kubernetes, and Skaffold. This approach deploys the complete OpenSCMS ecosystem in a single, coordinated workflow, including all SCMS services, their databases, and supporting infrastructure such as RabbitMQ.

For local development and evaluation, OpenSCMS provides a reference deployment based on Minikube.

Environment Setup Using Minikube

Installing Required Tooling: All required tooling should be installed using binenv, as described earlier. This ensures that consistent and compatible versions of all Kubernetes-related tools are used across different development environments.

Configuring Minikube: OpenSCMS uses the Docker driver for Minikube. This can be configured permanently by editing (or creating) the Minikube configuration file with the following content:

{
  "vm-driver": "docker"
}

Installing kubectl Plugins: To simplify cluster management, install the krew plugin manager for kubectl:

(
  set -x; cd "$(mktemp -d)" &&
  OS="$(uname | tr '[:upper:]' '[:lower:]')" &&
  ARCH="$(uname -m | sed -e 's/x86_64/amd64/' \
                         -e 's/\(arm\)\(64\)\?.*/\1\2/' \
                         -e 's/aarch64$/arm64/')" &&
  KREW="krew-${OS}_${ARCH}" &&
  curl -fsSLO "https://github.com/kubernetes-sigs/krew/releases/latest/download/${KREW}.tar.gz" &&
  tar zxvf "${KREW}.tar.gz" &&
  ./"${KREW}" install krew
)
echo 'export PATH="$HOME/.krew/bin:$PATH"' >> $HOME/.bashrc
exec $SHELL

Verify the installation:

kubectl krew --help

Finally, install the RabbitMQ plugin to simplify inspection and configuration of the message broker:

kubectl krew install rabbitmq

Starting Minikube and Deploying OpenSCMS

Start Minikube with:

minikube start

The first startup may take several minutes as container images are downloaded.

Once Minikube is running, perform a test deployment of OpenSCMS from the repository root:

skaffold run && skaffold delete

This command deploys the full SCMS stack and then immediately tears it down, verifying that all components can be built and scheduled correctly.

After the deployment completes, verify that the RabbitMQ components were installed successfully:

kubectl get ns

A namespace named rabbitmq-system should be present.

Accessing Services via Minikube

Minikube exposes services using an IP address that can be obtained with:

minikube ip

For convenience, this IP address can be mapped to a hostname by adding an entry to /etc/hosts:

sudo echo "$(minikube ip) minikube" >> /etc/hosts
ping minikube

This allows services to be accessed using a stable hostname during development and testing.
