Setup & Deployment

Certificates and Keys

All SCMS components require X.509 certificates and corresponding private keys to participate in the Public Key Infrastructure. These certificates are used for mutual TLS authentication, signing operations, and enrollment processes. Each service must have access to its certificate and private key, stored in PEM format, before it can be started.

OpenSCMS provides a certificate generation script for development and testing purposes. This script generates a complete PKI hierarchy including root certificates, intermediate certificates, and end-entity credentials for all services.

Generating Certificates for Development

From the repository root, run the certificate generation script:

./scripts/generate_certs.sh

This script generates a full set of test certificates and stores them in the certs/ directory within the repository. Each service will look for its certificate and key in this directory by default.

The script creates the following certificate structure:

  • Root CA: A self-signed root certificate that anchors the entire trust hierarchy.
  • Intermediate CAs: Certificates for each policy-specific intermediate authority (e.g., Enrollment CA, Authorization CA).
  • Service Certificates: End-entity certificates for each SCMS service (RA, LA, ECA, ACA) used for mutual TLS and API authentication.

Important: The certificates generated by this script are intended for development and testing only. They should never be used in production environments. Production deployments must use properly issued certificates from trusted Certificate Authorities.

Custom Certificate Configuration

For production deployments, custom certificates must be generated and configured. The expected certificate paths can be overridden using environment variables specific to each service. Refer to each service's README.md for details on configuring custom certificate paths.
