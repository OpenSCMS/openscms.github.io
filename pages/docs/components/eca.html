Enrollment Certificate Authority (ECA)

Overview

The Enrollment Certificate Authority (ECA) is responsible for issuing enrollment certificates, which establish the long-term cryptographic identity of an End Entity (EE) within the SCMS. Enrollment certificates bind sensitive device identity information to cryptographic material and serve as the foundation for all subsequent authorization and provisioning flows.

Within the OpenSCMS architecture, the ECA participates directly in the EE lifecycle by transitioning devices from the Registered state to the Enrolled state, and by issuing renewed enrollment certificates during successor enrollment. The ECA does not maintain EE state locally; instead, it relies on the Registration Authority (RA) as the authoritative source of lifecycle information. Before issuing any enrollment certificate, the ECA queries the RA to validate that the EE is in a state compatible with the requested operation.

The ECA exposes REST endpoints for enrollment-related operations, but their usage is tightly constrained by lifecycle and policy checks enforced in coordination with the RA. This design minimizes the ECA's exposure surface while preserving a clear separation of responsibilities between identity issuance and lifecycle control.

Initial Enrollment Certificate Flow

The initial enrollment flow is responsible for issuing the first enrollment certificate for an EE. This operation represents a critical trust boundary within the SCMS, as enrollment certificates embed long-term identity attributes that are later used to authorize all other certificate provisioning activities.

From a cryptographic perspective, the initial enrollment request establishes the first authenticated link between the End Entity (EE) and the SCMS. The enrollment request submitted by the client must be digitally signed using the EE's canonical private key. This key corresponds to the canonical public key previously provided during the EE registration process and stored by the Registration Authority (RA).

Upon receiving an enrollment request, the ECA does not directly trust the cryptographic identity asserted by the client. Instead, the ECA queries the RA to retrieve the canonical information associated with the requesting EE, including the canonical public key. Using this information, the ECA verifies the digital signature of the enrollment request, ensuring that the request originates from the registered EE and has not been tampered with.

If signature verification succeeds and all lifecycle and policy preconditions are satisfied, the ECA issues an explicit IEEE 1609.2.1 enrollment certificate. This certificate is signed by the ECA and binds the EE's canonical identity to a new cryptographic credential that will be used for all subsequent interactions with the SCMS.

The enrollment certificate represents the transition from an unauthenticated registration context to an authenticated operational context. Once issued, it becomes the primary credential used by the EE to authenticate future protocol messages. For example, when the RA receives an authorization certificate request, it first validates the enrollment certificate by verifying its signature using the ECA certificate. Only after the enrollment certificate is validated does the RA verify the digital signature of the incoming request using the public key contained in the enrollment certificate.

Enrollment Certificate Request

POST /v3/enrollment-certificate

Description:
  Submits a request for an initial enrollment certificate.

Input:
  An IEEE 1609.2.1 Enrollment Certificate Request SPDU containing canonical
  identity information and cryptographic material, signed using the EE's
  canonical private key.

Output:
  A response SPDU containing the newly issued enrollment certificate.

Errors:
  Requests originating from EEs in invalid lifecycle states, malformed
  SPDUs, signature verification failures, or cryptographic validation
  errors result in protocol-defined error responses.

Security and Trust Considerations

The ECA operates on highly sensitive identity data and is therefore designed to remain narrowly scoped. It does not perform EE registration, lifecycle persistence, or authorization policy enforcement directly. These responsibilities are delegated to the RA, which acts as the single source of truth for EE state and eligibility.

By enforcing strict preconditions through RA coordination and limiting direct client interaction, the ECA ensures that enrollment certificates are issued only under well-defined, auditable conditions, preserving the security and privacy guarantees of the SCMS.
