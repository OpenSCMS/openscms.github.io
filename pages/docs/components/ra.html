Registration Authority (RA)

Overview

The Registration Authority (RA) is the primary front-end component of OpenSCMS and the main interaction point for V2X clients. For architectural simplicity, the RA consolidates the roles of the Device Configuration (DC) and Device Configuration Manager (DCM) defined in IEEE 1609.2.1.

In practice, this makes the RA responsible for End Entity (EE) lifecycle management, distribution of trust and configuration artifacts, and coordination of certificate provisioning workflows with other SCMS components.

Internal Organization and Execution Model

The RA is internally organized into two deployment-time roles: the RA Frontend and the RA Worker. The RA Frontend exposes the external REST interface and handles all client-facing protocol logic, while RA Workers execute SCMS operations asynchronously.

The RA Frontend validates incoming SPDUs, verifies signatures, decrypts encrypted payloads when required, and constructs internal tasks that are enqueued for processing. It responds immediately to the client with an acknowledgment SPDU indicating when the client should retry the corresponding download request.

RA Workers monitor the task queue and execute tasks independently. They interact with internal SCMS services such as the ACA and ECA, perform cryptographic processing, and persist results to the database using the request hash as the primary correlation key.

Authorization Certificate Provisioning Flow

The Authorization Certificate provisioning flow is responsible for issuing batches of authorization (pseudonym) certificates that are used by V2X clients for authenticated application-layer communication.

This flow is intentionally split into two phases: a request phase and a download phase. The request phase is synchronous and lightweight, allowing the RA to validate and acknowledge the request quickly. The actual certificate generation is performed asynchronously to support high request volumes and batch-based issuance. The download phase allows the client to retrieve the generated certificates once processing has completed.

From an architectural perspective, the RA Frontend terminates the client connection, validates and decrypts the request, and enqueues a task for asynchronous execution. One of the RA Worker instances then processes the task by invoking the Authorization Certificate Authority (ACA), persists the generated certificates, and makes them available for download.

Authorization Certificate Request

The client submits an Authorization Certificate Request SPDU to the RA Frontend. The request contains cryptographic material and authorization parameters required to generate one or more authorization certificates.

POST /v3/authorization-certificate

Description:
  Submits a request for one or more authorization (pseudonym)
  certificates.

Input:
  - IEEE 1609.2.1 Authorization Certificate Request SPDU
  - Contains enrollment reference, butterfly key parameters,
    certificate permissions, and validity constraints

Output:
  - Acknowledgment SPDU (ACK)
  - Includes timing information indicating when the client
    should issue the download request

Errors:
  - Invalid or malformed SPDU
  - Signature verification failure
  - Decryption failure
  - Policy or parameter violations

Authorization Certificate Download

GET /v3/authorization-certificate

Description:
  Retrieves the batch of authorization certificates generated
  for a previously submitted request.

Input:
  - Download Request SPDU
  - Includes request hash returned during the request phase

Output:
  - Response SPDU containing a batch of authorization
    (pseudonym) certificates

Errors:
  - Unknown or expired request hash
  - Certificates not yet available

Successor Enrollment Certificate Flow

The Successor Enrollment Certificate flow allows an already enrolled End Entity to obtain a new enrollment certificate while preserving sensitive identity information. This flow is used to renew enrollment credentials without re-running the initial enrollment process.

As with authorization provisioning, the flow is divided into a request phase and a download phase. The request phase validates and decrypts the incoming enrollment data and schedules certificate generation asynchronously. The Enrollment Certificate Authority (ECA) performs the actual issuance of the new enrollment certificate, which is then stored by the RA and made available for download.

The RA acts as the cryptographic ingress point for this flow, decrypting the request and enforcing lifecycle and policy constraints before forwarding it to the ECA.

Successor Enrollment Certificate Request

POST /v3/successor-enrollment-certificate

Description:
  Submits a request for a successor enrollment certificate.

Input:
  - Encrypted IEEE 1609.2.1 Successor Enrollment Request SPDU
  - Contains identity-related enrollment parameters

Output:
  - Acknowledgment SPDU (ACK)
  - Indicates when the client should issue the download request

Errors:
  - Decryption failure
  - Invalid enrollment state
  - Malformed request SPDU

Successor Enrollment Certificate Download

GET /v3/successor-enrollment-certificate

Description:
  Retrieves the successor enrollment certificate generated
  for a previous request.

Input:
  - Download Request SPDU
  - Includes request hash

Output:
  - Response SPDU containing the new enrollment certificate

Errors:
  - Unknown request hash
  - Certificate not yet available

EE Registration and Policy Distribution

Before an End Entity (EE) can participate in enrollment or certificate provisioning flows, it must first be registered with the SCMS. In OpenSCMS, this initial step is handled by the Registration Authority (RA), acting in its combined DC/DCM role.

The registration process establishes the EE identity within the system, associates a canonical identifier with a device identifier, and returns the initial policy that governs which SCMS operations the EE is allowed to perform. This flow is synchronous and is typically executed during initial device provisioning or manufacturing.

EE Registration Request

POST /dcms/device/na

Description:
  Registers a new End Entity (EE) in the SCMS.

Input:
  - JSON payload containing EE metadata and identifiers

Output:
  - Confirmation of successful registration

Errors:
  - Duplicate EE registration
  - Invalid or incomplete metadata

The EE registration request is a JSON-based request that provides the RA with the minimal information required to identify and register a new device.

Request Payload Example:

{
  "deviceType": "obu",
  "publicKey": "04e9fed3f1a32e52ec...d0",
  "version": "na"
}

Semantics: The request identifies the type of EE (e.g., OBU or RSU) and provides a public key that will be associated with the device during its lifecycle. The version field allows compatibility handling across SCMS protocol versions.

Invalid payloads, unsupported device types, or malformed keys result in an error response and no state change.

EE Registration Response

Upon successful registration, the RA creates a new EE record and returns a registration response containing the identifiers assigned to the EE and the policy that governs its interaction with the SCMS.

Response Payload Example:

{
  "canonicalId": "1f81f9...49f225fb248c3",
  "deviceId": "1587425867723428852",
  "deviceType": "obu",
  "publicKey": "04e0991d6ab1e94....4d2e2616b1e8774b52cae53",
  "status": "Registered",
  "version": "na",
  "createdTime": "2025-12-26T13:59:31.627873478+00:00",
  "updatedTime": "2025-12-26T13:59:31.627882625+00:00",
  "devicePolicy": { ... }
}

Semantics: The response assigns a stable canonicalId, which is used across SCMS cryptographic workflows, and a deviceId, which uniquely identifies the EE within the SCMS database. The response also embeds the EE policy file that specifies which protocol endpoints and certificate flows the device is authorized to invoke.

At this stage, the EE is considered Registered but not yet Enrolled. Subsequent enrollment and certificate provisioning flows depend on the policy returned during registration.

Policy File Download

GET /dcms/policy/na

Description:
  Returns policy and configuration information applicable
  to the requesting EE.

Input:
  - EE identifier and context information

Output:
  - Policy definition governing allowed SCMS operations

The Registration Authority exposes a single endpoint for policy retrieval. The policy returned depends on whether the request identifies a specific End Entity (EE).

If the request includes either the canonicalId or deviceId query parameter, the RA returns a device-specific policy that defines which SCMS operations the EE is authorized to perform. If no EE-identifying parameters are provided, the RA returns the global SCMS policy, which defines system-wide configuration, trust anchors, and cryptographic constraints.

Device-Specific Policy

When the request includes canonicalId or deviceId, the RA returns the policy associated with that specific EE. This policy controls which protocol flows the EE may invoke, as well as operational constraints such as certificate limits and supported regions.

Example Request:

GET /dcms/device/na?canonicalId=1f81a656b75fa5df95eea79824a0cef7bd64bad3

Example Fields Returned:

{
  "ee-scmsV3-eeAuth": {
    "authorization-certificate-request": ["Enrollment"],
    "authorization-certificate-download": ["Enrollment"],
    "successor-enrollment-certificate-request": ["Enrollment"]
  },
  "ee-countCert": 1,
  "ee-heartbeatInterval": 0
  ...
}
Global SCMS Policy

When no EE-identifying query parameters are provided, the RA returns the global SCMS policy. This policy defines parameters and constraints that apply uniformly across all registered EEs, including cryptographic algorithms, trust anchors, endpoint locations, and protocol behavior.

Example Request:

GET /dcms/device/na

Example Fields Returned:

{
  "quorumCount": 3,
  "ctlSeriesId": "FFFFFFFFFFFFFF00",
  "cryptoAlgorithm": [ "nistP256", "nistP384", "sha256", "sha384" ],
  "certRootCa": { ... },
  "certEca": { ... },
  "certAca": { ... },
  "raUrl": "http://192.168.49.2:30080/v3/",
  "ecaUrl": "http://192.168.49.2:30085/v3/",
  "scmsV3-error": "Fine",
  "scmsV3-eeAuth": { ... }
}

Semantics: The policies files provides the authoritative configuration required by the EE to interact correctly with the SCMS, including cryptographic constraints, endpoint locations, timing parameters, and authorization rules. Clients are expected to cache and periodically refresh this file according to their operational needs.

Trust Artifact Distribution

The RA also exposes endpoints for distributing trust and configuration artifacts required by V2X clients.

Endpoints include:

  • GET /v3/ra-certificate
  • GET /v3/ca-certificate
  • GET /v3/ctl
  • GET /v3/ccf-ctl
  • GET /v3/crl
  • GET /v3/composite-crl-ctl

These endpoints return protocol-compliant artifacts such as CA certificates, Certificate Trust Lists (CTLs), and Certificate Revocation Lists (CRLs). Errors are reported using SCMS-defined error semantics.

Cryptographic Responsibilities and Trust Role

The RA is the only OpenSCMS component whose certificate includes a public encryption key intended for use by V2X clients. This design makes the RA the cryptographic ingress point for sensitive operations.

Consequently, the RA decrypts and validates the majority of client-originated messages and verifies digital signatures across most protocol flows. During startup, the RA loads its own certificates as well as the certificates of other SCMS components from the certs/ directory, enabling it to distribute trust material as part of its DC/DCM responsibilities.

End Entity Lifecycle Management

The Registration Authority (RA) is the authoritative component responsible for maintaining the lifecycle state of each End Entity (EE) within the SCMS. The lifecycle state represents the EE's progression through registration, enrollment, and certificate provisioning stages, and is used as a hard precondition for allowing or denying protocol flows.

Lifecycle state is explicitly stored and managed by the RA and is queried by other SCMS components to enforce protocol correctness. State transitions are never implicit and only occur as a result of successful execution of specific SCMS flows.

Lifecycle States

The following lifecycle states are supported:

  • Registered: The EE has been registered in the SCMS but does not yet possess a valid enrollment certificate.
  • Enrolled: The EE possesses a valid enrollment certificate and is eligible to request authorization certificates.
  • Provisioning: The EE is actively provisioning authorization certificates.
  • SuccessorEnrolled: The EE has successfully completed a successor enrollment flow and holds a renewed enrollment certificate.
  • Blocked: The EE is explicitly blocked and is not allowed to participate in any SCMS flows.
  • Deleted: The EE has been removed from active use and all SCMS interactions are denied.

Valid Lifecycle Transitions

The RA enforces the following valid lifecycle transitions:

  • Registered → Enrolled: Triggered after a successful enrollment certificate issuance by the ECA.
  • Enrolled → Provisioning: Triggered when the RA accepts an authorization certificate request from the EE.
  • Registered → SuccessorEnrolled: Triggered after a successful successor enrollment flow.
  • Provisioning → SuccessorEnrolled: Triggered after a successful successor enrollment flow while provisioning.
  • * → Blocked: Triggered by administrative action.
  • * → Deleted: Triggered by administrative action.

All other transitions are considered invalid and are rejected by the RA.

Lifecycle Enforcement Across SCMS Components

The lifecycle state maintained by the RA is consulted by other SCMS components as part of their flow validation logic. In particular:

Before issuing an enrollment certificate, the Enrollment Certificate Authority (ECA) queries the RA using:

GET /dcms/device/na

The ECA verifies that the EE is in the Registered state. Enrollment requests originating from EEs in any other state are rejected.

Similarly, when the RA receives an authorization certificate request, it checks that the EE is in the Enrolled state. Upon accepting the request, the RA updates the EE lifecycle state to Provisioning.

For successor enrollment flows, the RA verifies that the EE is either in the Registered or Provisioning state. Upon successful issuance of a successor enrollment certificate, the lifecycle state is updated to SuccessorEnrolled.

Lifecycle State Updates

Lifecycle transitions are persisted via the following endpoint:

PATCH /dcms/device/na

The request must identify the EE using the deviceId or canonicalId query parameter and provide the new lifecycle state explicitly in the request body.

Example Payload:

{
  "status": "Enrolled"
}

Error Handling and Invalid States

If a protocol request is received that is incompatible with the current lifecycle state of the EE, the request is rejected with a protocol-defined error response. Examples include:

  • Enrollment certificate requests from EEs not in the Registered state.
  • Authorization certificate requests from EEs not in the Enrolled state.
  • Any SCMS operation originating from an EE in the Blocked or Deleted state.

By centralizing lifecycle state management and enforcement within the RA, OpenSCMS ensures consistent protocol behavior, clear state transitions, and robust error handling across all SCMS components, even in distributed and horizontally scaled deployments.
