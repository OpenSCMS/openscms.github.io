Authorization Certificate Authority (ACA)

Overview

The Authorization Certificate Authority (ACA) is responsible for issuing authorization certificates, also referred to as pseudonym certificates, which are used by End Entities (EEs) to authenticate and sign V2X application-layer messages while preserving privacy.

In the OpenSCMS architecture, the ACA does not interact directly with clients. Instead, it is invoked exclusively by the Registration Authority (RA) as part of the authorization certificate provisioning flows. This design ensures that all client-facing validation, lifecycle checks, and policy enforcement are centralized in the RA, while the ACA remains narrowly focused on certificate generation.

As with other SCMS components, the ACA relies on the shared low-level cryptographic and protocol implementation provided by the C-based oscms-codecs-bridge. All certificate generation logic, including ASN.1 handling, IEEE 1609.2.1 encoding, and cryptographic operations, is performed via this shared library.

Role in Authorization Certificate Flows

Authorization certificate issuance is always initiated by the RA. When the RA receives an authorization certificate request from an EE, it validates the request, verifies the enrollment certificate, checks the EE lifecycle state, and extracts all parameters required for certificate issuance. The RA then constructs an internal task and forwards the relevant authorization issuance parameters to the ACA.

Before invoking the ACA, the RA interacts with the Linkage Authority (LA) to obtain the linkage values required for authorization certificate generation. These linkage values are used to enable misbehavior resolution and certificate revocation while preserving unlinkability during normal operation. Once the linkage values are available, the RA invokes the ACA to generate the authorization certificates.

The ACA generates the requested batch of authorization certificates and returns them to the RA. The RA persists the resulting certificates and makes them available to the client through the authorization download flow.

Explicit and Implicit Certificates

OpenSCMS supports the generation of both explicit and implicit authorization certificates, as defined by IEEE 1609.2.1.

Explicit certificates contain full public key material and signatures based on standard ECDSA over the P-256 curve. Implicit certificates rely on ECQV-style key derivation, significantly reducing certificate size by eliminating the need to transmit full public keys. Implicit certificates reduce the certificate size by approximately 64 bytes compared to their explicit counterparts.

The choice between explicit and implicit certificates is driven by policy and configuration parameters evaluated by the RA and passed to the ACA as part of the issuance request.

Butterfly Key Expansion Mechanisms

Authorization certificate issuance in OpenSCMS supports multiple butterfly key expansion mechanisms, which allow efficient generation of large batches of pseudonym certificates from compact input material.

The following butterfly mechanisms are supported:

  • Original Butterfly Keys (OBK)
  • Unified Butterfly Keys (UBK)
  • Compact Unified Butterfly Keys (CUBK)

In addition to the butterfly expansion mechanisms, OpenSCMS supports both unencrypted and encrypted non-butterfly variants. For encrypted inputs, the encryption key used to protect the butterfly material is the public encryption key of the RA. This ensures that only the RA is capable of decrypting sensitive key material received from the EE before it is forwarded for certificate generation.

Authorization Certificate Issuance Flow

During authorization certificate issuance, the ACA receives a request from the RA containing:

  • the selected certificate type (explicit or implicit),
  • the butterfly key expansion parameters,
  • linkage values obtained from the LA,
  • validity and permission constraints derived from policy.

Using this input, the ACA invokes the low-level C cryptographic engine to generate a batch of IEEE 1609.2.1 authorization certificates. All cryptographic operations, including key expansion, signature generation, and ASN.1 encoding, are performed within the C library to ensure consistency and correctness across components.

Once certificate generation completes, the ACA returns the resulting certificate batch to the RA. The ACA itself does not persist certificates or maintain client state.

Security and Architectural Considerations

The ACA is intentionally designed as a stateless and narrowly scoped service. It does not maintain EE lifecycle information, does not perform policy evaluation, and does not expose any externally accessible APIs. All trust decisions are delegated to the RA, which acts as the system's control plane.

By centralizing cryptographic implementation in a shared low-level library and limiting the ACA's responsibilities to certificate generation, OpenSCMS reduces implementation complexity, minimizes the attack surface, and facilitates auditing and reuse of the cryptographic core across all SCMS components.
