Authorization Certificate Authority (ACA)

Explicit and Implicit Certificates

OpenSCMS supports the generation of both explicit and implicit authorization certificates, as defined by IEEE 1609.2.1.

Explicit certificates contain full public key material and signatures based on standard ECDSA over the P-256 curve. Implicit certificates rely on ECQV-style key derivation, significantly reducing certificate size by eliminating the need to transmit full public keys. Implicit certificates reduce the certificate size by approximately 64 bytes compared to their explicit counterparts.

The choice between explicit and implicit certificates is driven by policy and configuration parameters evaluated by the RA and passed to the ACA as part of the issuance request.

Butterfly Key Expansion Mechanisms

Authorization certificate issuance in OpenSCMS supports multiple butterfly key expansion mechanisms, which allow efficient generation of large batches of pseudonym certificates from compact input material.

The following butterfly mechanisms are supported:

  • Original Butterfly Keys (OBK)
  • Unified Butterfly Keys (UBK)
  • Compact Unified Butterfly Keys (CUBK)

In addition to the butterfly expansion mechanisms, OpenSCMS supports both unencrypted and encrypted non-butterfly variants. For encrypted inputs, the encryption key used to protect the butterfly material is the public encryption key of the RA. This ensures that only the RA is capable of decrypting sensitive key material received from the EE before it is forwarded for certificate generation.

Authorization Certificate Issuance Flow

During authorization certificate issuance, the ACA receives a request from the RA containing:

  • the selected certificate type (explicit or implicit),
  • the butterfly key expansion parameters,
  • validity and permission constraints derived from policy.

Using this input, the ACA invokes the low-level C cryptographic engine to generate a batch of IEEE 1609.2.1 authorization certificates. All cryptographic operations, including key expansion, signature generation, and ASN.1 encoding, are performed within the C library to ensure consistency and correctness across components.

Once certificate generation completes, the ACA returns the resulting certificate batch to the RA. The ACA itself does not persist certificates or maintain client state.
