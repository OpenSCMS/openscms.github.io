Architecture Overview

System Roles & Data Stores

  • Registration Authority (RA): EE registration, authorization-certificate request/ack/download, and successor enrollment orchestration. Stores state in RA DB.
  • Enrollment Certificate Authority (ECA): Enrollment certificates and successor enrollment responses. Stores state in ECA DB.
  • Authorization Certificate Authority (ACA): Authorization certificate issuance. Stores state in ACA DB.
  • Linkage Authority (LA): PLV/LS operations for linkage values. Stores state in LA DB.

Each component maintains its own local database.

1. EE Registration (Fully Synchronous)

The system registers an "EE" entity, and it must be the first interaction between the client and OpenSCMS. The client sends the public key it requests to use as canonical information. Based on this public key, the server generates a canonical ID and device ID for this client.

Public API (RA):

  • POST /dcms/device/na — Returns canonicalId, deviceId, and device policy
  • GET /dcms/device/na — Lookup by canonicalId/deviceId
  • PATCH /dcms/device/na — Update device status

2. Enrollment Request (Synchronous)

The client initiates an enrollment process. The server handles request decoding, capturing the applicant's canonical ID for identity verification. Based on the canonical ID, the ECA verifies the message signature and creates the certificate.

Public API (ECA):

  • POST /v3/enrollment-certificate → Returns EcaEeCertResponseSpdu (binary C‑OER)

3. Successor Enrollment Request and Download (Asynchronous)

Handles successor enrollment for an already‑registered entity. This process is asynchronous and goes through the RA. The RA receives the request, creates a working task, and places it in the Celery task queue. An RA worker processes the request by forwarding it to the ECA.

Public APIs:

  • RA POST /v3/successor-enrollment-certificate — Returns acknowledgment
  • RA GET /v3/successor-enrollment-certificate — Download certificate
  • ECA POST successor-enrollment-certificate — RA to ECA communication

4. Authorization Certificate — Request & Download (Async)

This is the main flow of OpenSCMS. The client asks the server to create authorization certificates. The server immediately acknowledges and performs the work asynchronously via Celery workers. The process involves interaction between the ACA and LA components to create the linked values and generate the batch of certificates.

Public API (RA):

  • POST /v3/authorization-certificate — Returns RaEeCertAckSpdu with download time
  • GET /v3/authorization-certificate — Returns zip file with all generated certificates

System Design

The system is designed with modularity and scalability in mind. Asynchronous processing is handled via Celery task queues, allowing each component to operate independently while maintaining secure communication channels between all entities.
